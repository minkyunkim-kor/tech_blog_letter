- 본문 요약: AWS DEVCRAFT 해커톤에서 사조시스템즈가 ERP 회계(FI) 데이터를 활용하는 Agent를 Strands Agents SDK와 Kiro(SDD 기반 AI 코딩 어시스턴트)로 구축한 과정을 정리했다. Planner-Data-Excel/Mail Agent로 구성된 Graph 패턴에서 최종적으로 Agents-as-Tools 패턴(Supervisor가 하위 Agent를 도구처럼 호출)으로 전환했고, 대용량 데이터는 LLM에 직접 넘기지 않고 파일로 생성 후 경로만 반환하는 방식을 택했다. 영역별로 나눠 개발하다 세션 간 Context 공유(승인 상태, JWT 만료 처리)가 누락되어 겪은 문제와 해결 과정도 공유한다. 차대 균형 검증 30분→30초, 회계 데이터 분석 1~2시간→1분으로 단축될 것으로 기대한다.
- 주목할 point: "Supervisor Agent는 Orchestration 역할만 해야 한다"는 원칙과, 개발자 간 Context 공유 부재로 겪은 실패 사례(터미널 승인 vs Streamlit 세션 처리 불일치, JWT 만료 정보 누락)가 실무적으로 유용하다.

## [Amazon OpenSearch 3.3 업그레이드로 미리캔버스의 검색 성능 개선](https://aws.amazon.com/ko/blogs/tech/miricanvas-dual-vector-search-opensearch-3-3-upgrade-2/)
- 회사: AWS (고객사: 미리디)
- 본문 요약: 미리캔버스 검색 인프라 최적화 시리즈 2편. OpenSearch 2.19에서 끝내 풀지 못했던 "대량 업데이트 배치 직후 세그먼트 병합 폭발" 문제를, derived source(벡터 데이터를 _source에 중복 저장하지 않음)와 Lucene 10.3 기반 세그먼트 병합 개선이 있는 3.3으로 업그레이드해 해결했다. Rolling Upgrade 대신 신규 클러스터+도메인 스위칭 전략을 택해 무중단·안전 롤백을 확보했으며, 검증 결과 CPU -40%, 검색 레이턴시 -60%, 세그먼트 수 -37%를 확인했다. 프로덕션 적용 후 평균 레이턴시 -59%, 최대 레이턴시 -85%(175ms→26.7ms)를 달성했다.
- 주목할 point: 테스트 환경에서 Rolling Upgrade를 시도했다가 형태소 분석기 패치 미적용으로 클러스터 전체가 red 상태에 빠진 경험을 공유하며 왜 신규 클러스터 전략을 택했는지 설득력 있게 설명한다.
- keyword: #OpenSearch #벡터검색 #derivedsource #Lucene #검색성능

## [Amazon OpenSearch Service로 미리캔버스의 듀얼 벡터 검색 도입과 성능 최적화](https://aws.amazon.com/ko/blogs/tech/miricanvas-dual-vector-search-opensearch-3-3-upgrade-1/)
- 회사: AWS (고객사: 미리디)
- 본문 요약: 미리캔버스 검색 인프라 최적화 시리즈 1편. "검색어 의미"만 이해하는 단일 벡터로는 현재 편집 중인 디자인과 스타일이 맞지 않는 결과가 섞이는 문제를, 시맨틱+비주얼 벡터를 한 쿼리에서 동시에 쓰는 듀얼 벡터 검색으로 해결했다(Rescore 방식은 1차 후보군 품질 한계로 폐기). knn-filtering으로 BM25 선필터를 적용해 레이턴시를 107ms→13ms(88%↓)로 줄였고, 채택률도 9.02%p 향상됐다. 근본 병목은 쿼리 연산이 아니라 `_source`에 저장된 벡터 데이터로 인한 페이지 캐시 스래싱(메모리 부족→디스크 I/O 폭증)임을 규명하고, 범용 인스턴스(m7g)에서 메모리 최적화 인스턴스(r7g)로 전환해 IOPS(MAX) -34%, 레이턴시 -21%를 달성했다.
- 주목할 point: CPU는 안정적인데 IOPS만 폭증하는 지표를 보고 "메모리 부족→디스크 I/O"를 의심해 `_plugins/_knn/stats`와 OS 레벨 지표로 근본 원인을 추적한 디버깅 과정.
- keyword: #OpenSearch #듀얼벡터검색 #kNN #IOPS #검색인프라

## [Iceberg는 정말 싸고 빠를까? – 운영 데이터 5가지 벤치마크 솔직한 결과](https://blog.afinit.com/iceberg-vs-athena-pyspark-benchmarks)

- 주목할 point: 큰 테이블부터 옮기고 조인되는 모든 테이블에 동일한 시간 조건을 걸어야 효과가 커진다는 실전 팁, 그리고 Athena와 Spark가 같은 SQL에도 다르게 반응해 반드시 측정 기반으로 확인해야 한다는 결론.
