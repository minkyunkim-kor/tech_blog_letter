## [Redis와 Valkey를 해킹해 보자(Feat. CVE ID)](https://meetup.nhncloud.com/posts/416)
- 회사: NHN Cloud
- 본문 요약: 개인 연구로 Valkey/Redis의 RESP 인젝션 취약점을 발견해 CVE-2025-67733(CVSS 8.5)을 발급받은 과정을 공유한다. Lua 스크립트에서 `redis.error_reply()`로 만든 오류 테이블을 `error()`로 던지면, `luaPushErrorBuff()`의 `sdstrim()`이 문자열 양 끝의 `\r\n`만 제거하고 중간의 `\r\n`은 그대로 남겨, 오류 메시지 안에 숨긴 `\r\n+FAKE` 같은 페이로드가 RESP 프로토콜상 별개의 응답으로 파싱된다. 이를 이용해 다음 클라이언트 응답을 조작하는 "소켓 포이즈닝"과, 거대한 Bulk String 길이를 주입해 클라이언트를 무한 대기시키는 DoS 공격이 가능함을 분석했다. Valkey 9.0.2 이상, Redis 8.4.2 이상에서 패치되었다.
- 주목할 point: RESP 프로토콜, Lua 스크립트 실행 흐름, C 함수 레벨(`addReplyErrorSdsEx`→`_addReplyToBuffer`)까지 코드 경로를 따라가며 취약점을 규명한 분석 깊이.
- keyword: #Redis #Valkey #CVE #보안취약점 #RESP인젝션

## [[인프라를 소프트웨어처럼 4/5] plan은 동작을 모른다: 인프라를 테스트·재현한다는 것](https://flex.team/blog/2026/07/06/backend28)
- 회사: flex
- 본문 요약: `terraform plan`은 "무엇이 바뀌는지"는 보여주지만 "동작하는지"는 apply해야 알 수 있다는 문제의식에서, flex는 Pulumi+Kotlin으로 인프라를 헥사고날 아키텍처(spec=Port, 클라우드 구현=Adapter)로 작성하고 그 Adapter 자리에 로컬 kind/vcluster를 끼워 apply 전에 동작을 검증한다. 기존 AWS의 Terraform 자산은 그대로 두고 그린필드(다른 클라우드)부터 Pulumi로 시작했으며, network·cluster·identity·storage·bootstrap 5개 모듈을 동일 구조로 선언해 다른 클라우드에서도 같은 모양의 환경을 재현했다. `pulumi up` 이후 ArgoCD가 모든 앱을 Synced·Healthy로 맞출 때까지 폴링해 "선언"과 "동작" 사이를 검증한다.
- 주목할 point: 3화(Environment Variant, 브랜치로 환경을 찍어내는 것)가 두렵지 않을 수 있었던 전제가 바로 이 글의 "재현 가능성"이라는 연결고리를 짚는다.
- keyword: #IaC #Pulumi #헥사고날아키텍처 #Terraform #ArgoCD
