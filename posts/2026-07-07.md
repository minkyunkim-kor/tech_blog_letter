- 회사: 무신사(29CM)
- 본문 요약: 29CM Pricing 팀이 AWS Bedrock 기반 LLM 비용 증가의 원인을 찾기 위해, API별·토큰 타입별 사용량을 볼 수 있는 Prometheus/Grafana 메트릭 파이프라인을 먼저 구축했다. 그 결과 상품 속성 추출 API 하나가 전체 토큰의 약 92%를 차지하며, 15,000토큰짜리 시스템 프롬프트를 매번 동일하게 재전송하는 구조가 원인임을 확인했다. 시스템 프롬프트(고정)와 메시지 프롬프트(변동)를 분리한 뒤 Bedrock Converse API에 CachePointBlock(1시간 TTL)을 적용해 Prompt Caching을 도입했고, 1주일 운영 결과 캐시 히트율 98%, 전체 LLM 비용 64% 절감을 달성했다.
- 주목할 point: 비용 절감 옵션을 직관이 아니라 "코드베이스의 호출 패턴 + 가격표"를 입력으로 한 AI 비용 시뮬레이션으로 사전 검증하고, 최종 금액 계산만은 결정론적 경로(Grafana 집계치)에 맡겨 LLM이 숫자를 지어내지 못하게 분리한 점. 캐시가 깨지는 6가지 안티패턴(타임스탬프 삽입, 키 순서 비일관 등)도 실무적으로 정리했다.

## [OpenHTMLtoPDF 적용 경험](https://www.nextree.io/openhtmltopdf-jeogyong-gyeongheom/)
- 회사: 넥스트리
- 본문 요약: 견적서 등 문서를 PDF로 다운로드하는 기능을 만들면서 iText, OpenPDF, Apache PDFBox, OpenHTMLtoPDF 네 가지 PDF 라이브러리를 검토한 과정이다. iText는 상업적 라이선스 제약, OpenPDF는 스타일을 객체 단위로 작성해야 하는 유지보수 부담, PDFBox는 좌표 기반 레이아웃의 높은 개발 공수가 단점으로 꼽혔다. 최종적으로 HTML/CSS를 그대로 재활용할 수 있어 레이아웃 수정이 쉬운 OpenHTMLtoPDF를 채택했고, 한글 폰트 미지원 문제는 `builder.useFont()`로, 사용자 입력값으로 인한 HTML 구조 변조 위험은 `HtmlUtils.htmlEscape()`와 입력값을 리소스 참조 영역에서 배제하는 방식으로 대응했다.
- 주목할 point: 문서 양식이 자주 바뀌는 업무 특성상, PDF 생성 자체보다 "레이아웃을 얼마나 쉽게 수정할 수 있는가"를 최우선 선정 기준으로 삼은 점. 기능 비교뿐 아니라 사용자 입력이 HTML로 렌더링되는 구조에서 발생할 수 있는 보안 이슈까지 함께 다뤘다.
- keyword: #OpenHTMLtoPDF #PDF생성 #Java #보안
